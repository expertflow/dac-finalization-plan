# Action Items — DaC Pipeline Initiative

## 2. Brief Team Leads (This Week)
- Schedule a 30-minute call with each team lead across the 4–5 CX teams
- Do not wait for full manager sign-off on the product brief — the initiative brief is already approved and this can run in parallel
- Agenda: initiative overview, what the pipeline asks of developers, pilot team selection discussion
- Goal: no surprises when the pilot begins; team leads should feel informed, not announced to

## 3. Move to PRD After Manager Sign-Off
- Once the product brief is approved, initiate the PRD creation phase
- Bring in the architect to define technical requirements:
  - AI bot: which model/service, how it reads PR context, how drafts are structured
  - GitHub Actions pipeline design and trigger conditions
  - Same-repository doc file structure and naming conventions
  - Docusaurus publishing workflow (build + deploy on merge)
  - Documentation coverage tracking mechanism (per feature, per team)
  - Weekly team lead coverage report format and delivery
- PRD must be specific about pilot scope — repo, team, sprint, success threshold

## 4. Confirm Pilot Team Before PRD is Complete
- Identify and confirm the one CX team that will lead the pilot (two-week head start before others join)
- Selection criteria: team willingness, repo readiness, upcoming sprint with new feature work, team lead buy-in
- Lock this in before the PRD is finalized — the PRD will need to specify the pilot team's repo and sprint details
- Pilot team lead becomes the first point of feedback for pipeline refinement
